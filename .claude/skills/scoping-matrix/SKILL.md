# Scoping Matrix

> Reference for classifying project requests into structured scope templates.
> Each row is a pattern; actual scopes are customized from the pattern.
> This is a living document — add rows as new use-case patterns emerge.

---

## Structure

Each row represents a recurring project pattern. The columns are:

| Column | Purpose |
|---|---|
| **Specialization** | Top-level domain: Strategy, Research, Analysis, Operations, Finance, Marketing, Product, Tech |
| **Function** | Specific sub-area within the specialization (e.g., Competitive Intelligence under Strategy) |
| **Responsibility Category** | Type of work being done: Audit, Analysis, Synthesis, Evaluation, Design, Planning |
| **Use Case** | The concrete project pattern — short, plain-English name a founder would recognize |
| **Required Inputs** | What the client must provide before work can begin |
| **Deliverable Pattern** | What will be produced at the end of the engagement |
| **Acceptance Criteria Pattern** | How success is verified — what "done" means for this pattern |
| **Standard Exclusions** | What this engagement explicitly does NOT include |
| **Effort Estimate** | Realistic hours for a competent consultant (must fit within the ~10-hour cap) |
| **Suitability** | When this pattern is a good fit, and when it isn't |

## Seed Rows

### Row 1: Strategy — Competitive Landscape Analysis

| Field | Value |
|---|---|
| **Specialization** | Strategy |
| **Function** | Competitive Intelligence |
| **Responsibility Category** | Analysis |
| **Use Case** | Map the competitive landscape for a defined market segment |
| **Required Inputs** | Company overview (1–2 pages or deck), target customer definition, list of known competitors (minimum 3), geographic focus |
| **Deliverable Pattern** | Structured competitor matrix (positioning, pricing tier, key differentiators, strengths/weaknesses), written narrative summary, 2–3 strategic implications for the client |
| **Acceptance Criteria Pattern** | Matrix covers all provided competitors plus any material omissions the consultant identifies; each competitor has a complete row; strategic implications are specific to the client's stated position, not generic |
| **Standard Exclusions** | Primary research (customer interviews, surveys); win/loss interview programs; ongoing monitoring; go-to-market planning; pricing strategy recommendations |
| **Effort Estimate** | 6–9 hours |
| **Suitability** | Good fit when the client is pre-launch or early-stage and needs a clear picture of the field before making positioning decisions. Not a good fit when the market is highly technical/specialized and the consultant lacks domain depth — flag for skill matching. |

---

### Row 2: Research — Customer Discovery Synthesis

| Field | Value |
|---|---|
| **Specialization** | Research |
| **Function** | Customer Discovery |
| **Responsibility Category** | Synthesis |
| **Use Case** | Synthesize existing customer discovery interviews into actionable insights |
| **Required Inputs** | Raw interview transcripts or detailed notes (minimum 5 interviews), target customer persona description, the 2–3 core questions the discovery was designed to answer |
| **Deliverable Pattern** | Thematic synthesis document: key themes (with supporting quotes), validated vs. invalidated hypotheses, top 3–5 customer insights, recommended next questions |
| **Acceptance Criteria Pattern** | Every major theme is supported by at least 2 independent data points from separate interviews; hypotheses are explicitly marked validated/invalidated/inconclusive with evidence cited; no insight is presented without a direct reference to the underlying interview data |
| **Standard Exclusions** | Conducting new interviews; survey design; persona creation from scratch; quantitative research; product recommendations beyond what the data supports |
| **Effort Estimate** | 5–8 hours |
| **Suitability** | Good fit when the founder has done discovery but hasn't had time to synthesize it properly, or when an outside perspective on the patterns would be valuable. Not a good fit when fewer than 5 interviews exist — the sample is too thin for meaningful synthesis; redirect to interview guide design instead. |

### Row 3: Analysis — Unit Economics / Financial Model Review

| Field | Value |
|---|---|
| **Specialization** | Finance |
| **Function** | Unit Economics |
| **Responsibility Category** | Analysis |
| **Use Case** | Review and pressure-test an existing financial model or unit economics framework |
| **Required Inputs** | Financial model (spreadsheet), written explanation of key assumptions, current MRR/ARR (or revenue to date), pricing structure, primary customer acquisition channels and CAC estimates |
| **Deliverable Pattern** | Annotated model with comments flagging assumption risks, written review memo (1–2 pages) covering: model structure quality, assumption defensibility, key risks, and 3–5 specific recommendations to strengthen the model |
| **Acceptance Criteria Pattern** | Every flagged assumption includes a specific recommendation for how to support or validate it; review memo is self-contained (readable without the annotated model); recommendations are prioritized by impact |
| **Standard Exclusions** | Building a new model from scratch; financial forecasting or projections beyond what's in the existing model; accounting or tax advice; fundraising strategy; investor outreach |
| **Effort Estimate** | 5–8 hours |
| **Suitability** | Good fit when the client has a working model and needs a credibility check before a fundraise or board meeting. Not a good fit when no model exists — redirect to financial model build (a separate use case). Requires a consultant with direct SaaS/relevant business model experience. |

### Row 4: Operations — Process Audit and Bottleneck Identification

| Field | Value |
|---|---|
| **Specialization** | Operations |
| **Function** | Process Improvement |
| **Responsibility Category** | Audit |
| **Use Case** | Audit a defined operational process to surface bottlenecks and inefficiencies |
| **Required Inputs** | Written description of the current process (or process map if one exists), data on process volume and cycle times (even rough estimates), list of known pain points from the team, access to one 60-minute working session with the process owner |
| **Deliverable Pattern** | Current-state process map (if not provided, created as part of the engagement), bottleneck analysis identifying the top 3–5 friction points with root cause notes, prioritized recommendations with effort/impact sizing |
| **Acceptance Criteria Pattern** | Process map accurately reflects the current state as confirmed by the process owner; each bottleneck is tied to a measurable symptom (time, error rate, volume cap); recommendations include a "quick win" (implementable in under 1 week) and at least one structural fix |
| **Standard Exclusions** | Implementation of recommended changes; software selection or procurement; team restructuring recommendations; cross-departmental process redesign beyond the defined scope; change management |
| **Effort Estimate** | 7–10 hours |
| **Suitability** | Good fit when a specific process is visibly broken and the team knows it but hasn't had time to dig in. Not a good fit when the client wants broad organizational improvement — that's a consulting engagement beyond the 10-hour cap. Scope to one process. |

### Row 5: Marketing — Messaging and Positioning Audit

| Field | Value |
|---|---|
| **Specialization** | Marketing |
| **Function** | Brand and Messaging |
| **Responsibility Category** | Audit |
| **Use Case** | Audit current messaging and positioning for clarity, differentiation, and audience fit |
| **Required Inputs** | Current website (URL), any existing positioning or messaging docs, 2–3 example sales or marketing assets (emails, one-pagers, pitch deck), target customer description, 1–3 competitors the client benchmarks against |
| **Deliverable Pattern** | Audit report covering: clarity assessment (does the ICP instantly understand the value?), differentiation assessment (how well does messaging distinguish from competitors?), consistency check across assets, top 5–7 specific, prioritized recommendations with example rewrites for the highest-priority items |
| **Acceptance Criteria Pattern** | Every recommendation includes a specific example (rewrite, restructure, or alternative framing); audit covers all provided assets; recommendations are tied to the client's stated ICP, not generic best practices |
| **Standard Exclusions** | Brand identity or visual design; full website copywriting; SEO strategy; campaign creation; market research to validate positioning claims |
| **Effort Estimate** | 5–7 hours |
| **Suitability** | Good fit for early-stage companies whose messaging grew organically and hasn't been systematically reviewed. Also useful pre-fundraise or pre-launch. Not a good fit if the client wants new positioning built from scratch — that requires discovery and is a different (longer) engagement. |

### Row 6: Product — User Research Synthesis

| Field | Value |
|---|---|
| **Specialization** | Product |
| **Function** | User Research |
| **Responsibility Category** | Synthesis |
| **Use Case** | Synthesize user research (usability sessions, support tickets, NPS responses, or similar) into product priorities |
| **Required Inputs** | Raw research artifacts (session recordings, notes, or transcripts — minimum 5 sources), the product area or feature set in scope, the primary product question the research was meant to answer, current product priorities or roadmap (brief) |
| **Deliverable Pattern** | Insight report: top user needs and pain points (with evidence), usability issues ranked by severity and frequency, product recommendations mapped to the stated priorities, open questions flagged for follow-up research |
| **Acceptance Criteria Pattern** | Each insight is supported by at least 2 independent data points; severity rankings are explicit (critical / significant / minor); recommendations are scoped to what's achievable within the existing product area; no speculation presented as finding |
| **Standard Exclusions** | Conducting new research sessions; survey or research instrument design; UX design or wireframes; implementation; roadmap sequencing decisions (client retains those) |
| **Effort Estimate** | 5–8 hours |
| **Suitability** | Good fit when a product team has research artifacts they haven't had bandwidth to synthesize. Also effective when an outside read on patterns would reduce internal bias. Not a good fit when research artifacts don't exist yet — redirect to research planning. |

### Row 7: Tech — Vendor / Tooling Evaluation

| Field | Value |
|---|---|
| **Specialization** | Tech |
| **Function** | Technology Selection |
| **Responsibility Category** | Evaluation |
| **Use Case** | Evaluate and recommend a tool or vendor in a defined category (e.g., CRM, CDP, data warehouse, analytics platform) |
| **Required Inputs** | Defined category and problem statement, list of requirements (must-have vs. nice-to-have), current tech stack summary, 2–5 vendors/tools already under consideration, budget range, team size and technical capability level |
| **Deliverable Pattern** | Evaluation matrix (vendors vs. requirements), written recommendation with rationale, risk notes for the top choice, implementation considerations (migration complexity, integration lift, estimated ramp time), runner-up notes |
| **Acceptance Criteria Pattern** | All provided must-have requirements scored for each vendor; recommendation is clearly stated (not "it depends"); risks are specific to the client's stack and team, not generic; a clear second choice is included in case the top recommendation doesn't move forward |
| **Standard Exclusions** | Vendor negotiation or procurement; RFP process management; implementation or migration; custom integration development; ongoing vendor management |
| **Effort Estimate** | 5–8 hours |
| **Suitability** | Good fit when a team is stuck in analysis paralysis on tooling or needs an expert opinion to move forward. Not a good fit for highly bespoke enterprise infrastructure decisions that require deep architectural context beyond what can be shared in 10 hours. |

### Row 8: Finance — Fundraising Narrative and Pitch Deck Review

| Field | Value |
|---|---|
| **Specialization** | Finance |
| **Function** | Fundraising |
| **Responsibility Category** | Evaluation |
| **Use Case** | Review and strengthen a seed or Series A pitch deck and fundraising narrative |
| **Required Inputs** | Current pitch deck (all slides), one-paragraph summary of what the round is for and target raise amount, investor type being targeted (angels, seed funds, Series A VCs), any written investor feedback already received |
| **Deliverable Pattern** | Annotated deck with slide-by-slide feedback, written review memo covering: narrative arc assessment, strength of the problem/solution framing, team slide credibility, traction framing, ask clarity, and top 5 priority revisions with specific rewrites or structural changes |
| **Acceptance Criteria Pattern** | Every slide with a recommended change includes a specific suggestion (not just "strengthen this"); narrative assessment explicitly addresses whether the deck tells a coherent story; priority revisions are ordered, not listed as equally important |
| **Standard Exclusions** | Investor introductions or warm intros; financial modeling; legal review; design or visual polish; term sheet negotiation; full deck rebuild from scratch |
| **Effort Estimate** | 4–7 hours |
| **Suitability** | Good fit in the 2–4 weeks before a fundraise kicks off, or when a round has stalled and the founder suspects the narrative is the issue. Not a good fit if the company has no traction to report and no clear hypothesis for why investors should care — the story can't be polished if the substance isn't there yet. Requires a consultant with direct fundraising or VC-side experience. |

## How to Use

### 1. Classify an incoming project request

When a client submits a project request (via the intake form), read the request description and answer three questions:

1. **What domain is this?** Match to a Specialization (Strategy, Research, Analysis, Operations, Finance, Marketing, Product, Tech).
2. **What type of work is being asked for?** Match to a Responsibility Category (Audit, Analysis, Synthesis, Evaluation, Design, Planning).
3. **What concrete output is expected?** Look for signals that match a Use Case name or Deliverable Pattern.

### 2. Select the best-fit row

Scan seed rows for the closest match. Selection criteria:

- The Use Case name should match the request's core intent, even if the wording differs.
- The Required Inputs should be things the client can reasonably provide.
- The Effort Estimate should fit within the ~10-hour cap without heroics.
- The Suitability notes should not disqualify this client's situation.

If no row is a close match, flag for admin review. Do not force a poor fit — it is better to classify as "custom" and route to an admin who can draft a scope manually.

### 3. Pre-fill scope fields

Once a row is selected, use it to pre-populate the draft scope:

- Copy the **Deliverable Pattern** as the starting point for the deliverable description.
- Copy the **Acceptance Criteria Pattern** and adjust for any client-specific details.
- Copy the **Standard Exclusions** verbatim into the exclusions field.
- Use the **Effort Estimate** range as the basis for hours; set the fee accordingly.
- Pull **Required Inputs** into the "what client must provide" section of the scope.

All pre-filled fields are editable by admin before the scope is approved. The matrix provides a starting point, not a final answer.

### 4. Validation before approval

Before an admin approves a scope draft (whether AI-assisted or manually written):

- Confirm the Required Inputs are realistic for this specific client.
- Confirm the Effort Estimate is appropriate given any client-specific complexity.
- Check that Standard Exclusions don't accidentally exclude something the client obviously expects to receive.
- Confirm Suitability notes don't flag a mismatch that wasn't caught in classification.

Human approval is required before any scope is shown to the client. See the `ai-gates` skill for the full gate table.
